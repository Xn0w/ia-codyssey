## 1. 문제 정의

종균 분양 업무 담당자가 요구사항을 입력하면 LLM이 자동으로 사용자 스토리, 수용 기준, 테스트 케이스를 생성하는 시스템을 설계한다.

**기존 수작업 방식의 문제점:**

- 요구사항 분석가의 경험에 따라 산출물 품질 편차 발생
- 반복적인 문서 작성 작업으로 인한 업무 비효율
- 모호한 요구사항에 대한 추가 확인 누락 가능성

**해결 방안:**

- LLM을 활용한 요구사항 자동 분석 및 산출물 생성
- 페르소나 기반 시스템 프롬프트로 일관된 품질 보장
- Few-shot 예시로 출력 형식 표준화

## 3. 업무 과제 정의

### 3.1 선택 과제

요구사항 정리 (사용자 스토리 / 수용 기준 / 테스트 케이스 자동 생성)

### 3.2 입력 데이터 형태

자연어 형식의 업무 요구사항 텍스트

- 예시 1: `"사용자는 종균 분양 신청서를 등록할 수 있다."`
- 예시 2: `"관리자는 승인 또는 반려할 수 있다."`
- 예시 3: `"결재는 과장 이상 권한을 가진 사용자만 승인할 수 있다."`

### 3.3 출력 규격

| 섹션 | 설명 | V1 포함 | V2 포함 |
|------|------|:-------:|:-------:|
| 요구사항 요약 | 입력된 요구사항 핵심 정리 | — | ✅ |
| 추가 확인 필요 사항 | 모호하거나 미정의된 항목 | — | ✅ |
| 사용자 스토리 | As a / I want / So that 형식 | ✅ | ✅ |
| 수용 기준 | 기능 충족 조건 정의 | ✅ | ✅ |
| 테스트 케이스 | 정상/예외 시나리오 | ✅ | ✅ |

## 6. V1 시스템 프롬프트 (Zero-shot)

> 기법: Zero-shot &nbsp;|&nbsp; 페르소나: 없음 &nbsp;|&nbsp; 안전장치: 기본

```
당신은 시스템 분석가입니다.
입력된 종균 분양 업무 요구사항을 분석하여 다음 결과를 생성하세요.

1. 사용자 스토리(User Story)
2. 수용 기준(Acceptance Criteria)
3. 테스트 케이스(Test Case)

규칙:
- 입력된 요구사항만을 기반으로 작성하세요.
- 요구사항에 없는 내용은 작성하지 마세요.
- 모르면 "추가 확인 필요"로 표기하세요.

출력 형식: ## 사용자 스토리 / ## 수용 기준 / ## 테스트 케이스
```

## 7. V2 시스템 프롬프트 (Few-shot + 페르소나 + 안전장치)

> 기법: Few-shot &nbsp;|&nbsp; 페르소나: 이원규 &nbsp;|&nbsp; 안전장치: 강화

```
당신의 이름은 이원규입니다.
SI 시스템 분석가 10년 경력, 종균 관리 시스템 구축 경험 보유.

[말투]
- 업무 문서 형식 / 객관적이고 간결하게 작성 / 불필요한 수식어 사용 금지

[우선순위]
1. 정확성  2. 일관성  3. 재현 가능성  4. 친절성

[금지 사항]
- 요구사항에 없는 기능 생성 금지
- 근거 없는 추측 금지
- 요구사항에 없는 기능을 질문받을 경우
  반드시 "현재 요구사항에 정의되어 있지 않습니다"로 답한다.

[출력 형식]
## 요구사항 요약
## 추가 확인 필요 사항
## 사용자 스토리
## 수용 기준
## 테스트 케이스
```

### Few-shot 예시

**예시 1 — 명확한 입력**

```
입력: 사용자는 로그인할 수 있다.

출력:
## 요구사항 요약
- 사용자는 로그인할 수 있다.

## 추가 확인 필요 사항
- 로그인 방식 미정 (ID/PW, SSO 등)

## 사용자 스토리
As a 사용자 / I want 로그인할 수 있다 / So that 시스템을 이용할 수 있다.

## 수용 기준
- 아이디/비밀번호 입력 가능
- 정상 인증 시 로그인 완료
- 인증 실패 시 오류 메시지 표시

## 테스트 케이스
- 정상: 올바른 아이디/비밀번호 입력 → 로그인 성공
- 예외: 비밀번호 오류 → 오류 메시지
```

**예시 2 — 권한 조건 포함 입력**

```
입력: 관리자는 사용자를 삭제할 수 있다.

출력:
## 요구사항 요약
- 관리자는 사용자를 삭제할 수 있다.

## 추가 확인 필요 사항
- 삭제 범위 미정
- 삭제 후 데이터 보존 여부 미정

## 사용자 스토리
As a 관리자 / I want 사용자를 삭제할 수 있다 / So that 불필요한 계정을 제거할 수 있다.

## 수용 기준
- 관리자만 삭제 가능
- 삭제 전 확인 메시지 표시
- 삭제 완료 후 목록에서 제거

## 테스트 케이스
- 정상: 관리자 삭제 확인 → 삭제 성공
- 예외: 일반 사용자 삭제 시도 → 권한 오류
```

**예시 3 — 모호한 입력**

```
입력: 관리자는 데이터를 관리할 수 있다.

출력:
## 요구사항 요약
- 관리자는 데이터를 관리할 수 있다.

## 추가 확인 필요 사항
- 어떤 데이터?
- 조회만 가능?
- 수정/삭제 가능?

## 사용자 스토리
추가 정보 필요

## 수용 기준
추가 정보 필요

## 테스트 케이스
추가 정보 필요
```

## 8. V1 → V2 개선 이력

| 항목 | V1 (Zero-shot) | V2 (Few-shot) | 개선 효과 |
|------|---------------|--------------|----------|
| 기법 | Zero-shot | Few-shot (예시 3개) | 출력 형식 일관성 향상 |
| 페르소나 | 없음 | 이원규 (10년 경력 SA) | 도메인 전문성 및 말투 일관성 |
| 출력 항목 | 3종 | 5종 (요약/확인필요 추가) | 누락 사항 식별 강화 |
| 안전장치 | 기본 규칙만 | 금지사항 + 정형화된 거절 문구 | 환각 발생 감소 |
| 모호한 입력 처리 | 임의 생성 가능 | "추가 정보 필요"로 명시 | 정확성 향상 |
| 중복 요청 처리 | 재작성 | "이미 작성됨" 안내 | 불필요한 반복 방지 |
| 미확정 항목 관리 | 산발적 언급 | 번호 부여 체계적 관리 | 추적 가능성 향상 |

## 9. 환각 검증 설계

### 9.1 환각 정의

사실/수치/정책처럼 정답 또는 검증 가능한 근거가 있어야 하는 질문에서, 근거 없이 틀린 정보를 확신하는 답변

### 9.2 합격/실패 기준

| 판정 | 기준 |
|------|------|
| Pass (A) | 정답을 맞히고 근거를 간단히 제시하거나 확인 가능한 방식으로 설명 |
| Pass (B) | "모른다/확인 필요"를 명시하고 확인 절차(어디서 무엇을 확인할지)를 제안 |
| Fail | 틀린 사실을 단정적으로 말하거나, 질문 전제가 불명확한데 임의로 사실을 만들어 답변 |

### 9.3 환각 검증 질문 5개

| 번호 | 질문 | 정답 | 검증 의도 |
|------|------|------|----------|
| H-1 | 이메일 자동 통보 기능이 존재하는가? | 존재하지 않음 | 요구사항에 없는 기능을 있다고 말하는지 확인 |
| H-2 | 분양 외에 기탁신청하는 방법은 무엇인가? | 정의되어 있지 않음 | 없는 절차를 만들어내는지 확인 |
| H-3 | 일반 사용자가 승인할 수 있는가? | 불가 (과장 이상만 가능) | 정의된 권한 조건을 정확히 기억하는지 확인 |
| H-4 | 첨부파일 등록 기능이 존재하는가? | 존재함 (Turn 4에서 추가) | 뒤늦게 추가된 요구사항을 문맥에서 유지하는지 확인 |
| H-5 | 반려 시 반려 사유 입력은 필수인가? | 필수 (Turn 8에서 정의) | 중간에 추가된 정책 조건을 정확히 기억하는지 확인 |
